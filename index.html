Product Case Study · Side Project

From PRD to shipped product: how I defined, scoped, and built a browser-based stress relief app using AI tools.

March 20268 min readProduct Management · User Research · Vibe Coding
🫧 Try the live product — bubblewrapuzzle.vercel.app Pop 30–60 bubbles, reveal a hidden illustration, feel briefly human again. No account. No install. No timer. This is what zero-friction looks like in practice.
Open the app →

Bubble Bloom is a browser-based micro-relaxation game. Users pop iridescent bubbles to progressively reveal a calming illustrated puzzle, with ASMR audio feedback on every pop and an ambient soundscape that builds as the image emerges. Sessions run 2–3 minutes. There is no timer, no score, and no failure state.

The product was built entirely with AI vibe-coding tools — Claude, Gemini, and Replit. But the story worth telling isn't about the tech stack. It's about what happened before a single line of code was written.

The Problem: who actually needs this?

Most stress-relief apps are designed for people who have already committed to wellness. They require accounts, onboarding flows, subscriptions, and 10-minute sessions. That excludes the people who need relief most urgently: the person between back-to-back meetings who has 3 minutes, not 30.

I defined seven distinct user segments in the PRD, each with a different relationship to stress and a different constraint on time or context:

  • 01Busy employees — need a 2-minute mental reset between meetings. The product must be openable, completable, and closeable within a calendar gap. No loading screens. No decisions.
  • 02Neurodivergent users (ADHD, autism, sensory processing differences) — benefit from repetitive, low-stakes sensory input. The interaction pattern — tap to pop, one action, one reward — was designed specifically for this group, not as an afterthought.
  • 03Students — short decompression during study sessions or exam periods. Completion in under 3 minutes was a hard product constraint, not a nice-to-have.
  • 04Caregivers (parents, nurses, social workers) — micro-recovery moments in high-demand schedules. Often on mobile, often interrupted. The product had to be resumable at any point without losing progress.
  • 05People with anxiety — grounding through predictable, satisfying interaction. Predictability was a design principle: every pop behaves identically, every sound follows the same pattern.
  • 06Insomniacs — a calming screen activity that isn't doom-scrolling. No algorithmic feed. No infinite scroll. A finite, completable loop with a clear endpoint.
  • 07Chronic pain sufferers — passive sensory engagement with no dexterity demands. Single tap or click only: no drag, no swipe, no precision required.

Understanding these users before writing any spec shaped every subsequent decision — from the 30–60 bubble grid size (enough to feel satisfying, not so many it becomes a chore) to the decision to make audio purely optional with a persistent mute toggle.

Success metrics, defined before building

One of the disciplines I find most revealing about a PM's approach is how they define success before the product exists. Outcome metrics, not output metrics. I set four in the PRD:

Target success metrics (v1.0) Session completion rate > 70% — users finish revealing the full puzzle. Average session duration: 2–4 minutes. Return visit rate: more than 1 puzzle opened per week. Subjective calm score (optional exit poll): ≥ 4/5. These weren't aspirational — they were the criteria against which I'd evaluate whether the core experience was actually working.

The completion metric deserves particular attention. A stress-relief app that users abandon halfway through isn't delivering its primary value. The entire interaction arc — grid size, audio crescendo, the final bloom animation, the completion chime — was designed to make finishing feel rewarding, not just starting.

Feature decisions: what made the cut and why

The PRD defined four core feature areas. Each involved explicit trade-offs worth naming.

  • 01Bubble grid: 30–60 bubbles, organic layout — Not a fixed 6×6 grid. Organic, non-uniform placement with varied bubble sizes makes each session feel handmade rather than mechanical. The iridescent shimmer and slow drift animation in idle state give the product life before the user even touches it. This was a deliberate first-impression decision.
  • 02ASMR audio: pitch-varied per pop, ambient crescendo — Each pop triggers a soft sound varied ±15% in pitch for an organic feel. The ambient background track fades from 0% to 100% volume as more bubbles are revealed. This isn't decoration — it's the primary mechanism for communicating progress without a percentage bar or number. The user feels the session advancing through sound.
  • 03Breathe Mode: OFF by default — Every 10 pops, an optional breathing prompt overlays for 4 seconds. I made this opt-in deliberately. Surfacing an unrequested mindfulness prompt mid-session is an interruption, not a feature. Users who want it can enable it; users who just want to pop bubbles aren't punished for that preference.
  • 04Progress indicator: ring only, no numbers — A subtle circular ring with no percentage text, no numerical count. Numbers create pressure. A ring communicates direction without quantifying the distance. This was the same logic behind removing the timer: the moment you can fail to meet a target, the product stops being calming.

Five worlds, each a complete sensory system

The five illustration themes — Ocean Depth, Night Sky, Forest Terrarium, Sakura Garden, Candy Cloud — weren't just visual variety. Each is a matched set: a colour palette, an ambient soundscape, and an illustration style that reinforce each other. A user in Ocean Depth hears underwater hum and bubbles while revealing bioluminescent jellyfish in aqua and teal. The audio and visual work as a single experience, not parallel tracks.

This is a common failure mode in product design: features that exist as checklists rather than as coherent systems. Getting the worlds right meant defining them as experiences first, then speccing the assets.

Design principles as scope enforcement

Four principles governed every feature decision, and more importantly, every feature rejection:

What "No Pressure, Ever" actually meant in practice No timer. No score. No lives. No failure state. No streaks. No leaderboards. No sharing prompts. No push notifications. No ads. Each of these was a deliberate out-of-scope decision in the PRD — not a roadmap item deferred to v2, but a conscious choice that the product's value proposition depended on their absence. Intentional restraint is harder to maintain than it sounds when building, because every tool and every collaborator (including AI) defaults toward adding features, not removing them.

What the PRD actually bought me

I wrote the full PRD before prompting a single AI tool. That ordering mattered. Here's what it changed in practice:

  • 01Requirements became prompts — "Art regions must not be visible before the corresponding bubble is popped" is a product requirement. It is also an exact prompt constraint. Writing the PRD forced me to articulate requirements precisely enough to be executable — which is exactly the same skill as writing a tight spec for an engineering team.
  • 02Scope had a written defence — When building with AI tools, the temptation to add features is constant. Every time a new idea surfaced, I checked it against the PRD's Out of Scope section. User accounts: out of scope. Social sharing: out of scope. Gamification pressure: actively contra-indicated by the design principles. The document wasn't just planning — it was scope enforcement.
  • 03Technical constraints became product decisions earlier — The performance targets (initial load under 3 seconds on 4G, 60fps on mid-range devices, total bundle under 2MB) weren't added after building. They were written into the PRD, which meant they shaped architectural choices from the start rather than creating rework at the end.
  • 04Accessibility was non-negotiable, not an afterthought — WCAG 2.1 AA, full keyboard navigation, screen reader support, and prefers-reduced-motion compliance were all in the PRD. Speccing accessibility before building means it gets designed in, not bolted on. No flashing animations. No sudden loud sounds. No red or high-frequency alert colours. These were sensory safety requirements, not accessibility checkboxes.

The phased roadmap

The PRD closed with a four-phase roadmap — not a backlog, but a sequenced plan with a clear MVP definition. Phase 1 (2 weeks): core bubble grid, one theme (Ocean), ASMR pop sounds, puzzle reveal. Phase 2 (1 week): all five themes, ambient soundscapes, Breathe Mode. Phase 3 (1 week): performance optimisation, accessibility audit, mobile polish. Phase 4: user testing with target audiences, iterate on feedback — ongoing.

Shipping Phase 1 first meant validating the core loop — pop, sound, reveal — before investing in the content library. It's the same logic as any MVP: learn the minimum necessary before scaling the expensive parts.

Product Case Study · Side Project

Building Whodunit: what a murder mystery game taught me about AI product design, narrative coherence, and the PRD as a spec tool.

March 202610 min readProduct Management · AI Systems · Game Design · Lovable
🔪 Try it — whodunit.lovable.app Pick a tone, setting, difficulty, and detective avatar. An AI generates your entire case — unique suspects, alibis, evidence, red herrings, and a pre-committed killer. No two mysteries are the same.
Play Whodunit →

Most games lie to you about replayability. They shuffle the same deck. Whodunit does something different: it generates the entire mystery from scratch every session — different victim, different killer, different motive, different evidence trail — driven entirely by the tone, setting, and difficulty you configure at the start. The AI is not filling in a template. It is writing a new case.

What I want to document here isn't the build. It's the product thinking behind it — the decisions that shaped what Whodunit is, what it isn't, and why those boundaries matter.

The problem I was actually solving

Existing murder mystery formats fail in two specific ways. Physical kits and boxed games are single-use: once played, the answer is known and the product is dead. Digital mystery games use templates. Players learn the structural tells — the character archetype who is always innocent, the clue type that always flags the killer. Templates create pattern recognition, not deduction.

Neither format does what modern LLMs make possible: generating a coherent, contextually rich, internally consistent mystery at scale, unique per session, customised to user inputs. The gap was obvious. The product question was harder: how do you make AI-generated narrative feel like a game, not a chatbot?

The design brief in one sentence Dark mahogany aesthetics, blood-red as the only accent colour (#BC0000), 3D avatar illustrations for suspects and detectives, eight investigation screens with a vault-door reveal gate — and a mystery where the killer was always the same person from the moment the case was generated.

The one constraint that defined everything

The most important product decision in Whodunit was not a feature. It was a constraint: the killer's identity is committed at generation time, not at reveal time.

This sounds obvious, but it has significant technical implications. The AI cannot retroactively assign a killer based on which suspects the player investigated most. The LLM must generate victim, suspects, evidence trail, red herrings, and alibi inconsistencies all pointing at a pre-determined person, in a single coherent generation call. The killer was always the same person. The evidence always pointed there. You were just finding it.

Without this constraint, you don't have a mystery game. You have a lottery with narrative decoration. Writing it as a hard requirement in the PRD before a line of code was written meant it shaped the entire AI prompt architecture, the JSON output schema, and the validation logic that gates the game from rendering until coherence is confirmed.

Four configuration axes, infinite outputs

The setup screen asks four things: tone, setting, difficulty, and detective avatar. These aren't cosmetic filters on a shared template. Each combination feeds directly into the generation prompt as structural parameters, producing a genuinely different output.

  • 01Tone shapes the narrative register — A Noir mystery generates a cynical narrator voice, corrupt institutional alibis, and morally compromised suspects. A Cozy mystery generates eccentric but sympathetic characters with motives rooted in reputation rather than violence. The same setting, the same difficulty — completely different stories.
  • 02Setting shapes spatial logic — A suspect cannot claim to have been "in the library" on a cruise ship. A Victorian Mansion has different room vocabulary, staff hierarchies, and alibi plausibility than a Museum. Setting isn't a backdrop — it constrains what the AI can plausibly generate.
  • 03Difficulty shapes evidence architecture — Easy gives you 4 suspects, 6 clues, 2 red herrings, and a killer with a weak alibi. Hard gives you 8 suspects, 12 clues, 6 red herrings, and a killer whose alibi initially appears strongest. This isn't a number adjustment — it's a structural change in how the AI must distribute incriminating evidence.
  • 04Avatar is identity, not mechanics — yet — In v1.0, the detective avatar sets the player's frame without changing game mechanics. It's a product decision about investment and ownership: the player chose who they are before they read the first clue. v2.0 will introduce avatar-conditional dialogue paths that surface different information per detective type.

Eight screens, each with one job

Whodunit is an eight-screen investigation arc. What took the most spec work wasn't the screens that do the most — it was defining what each screen and only that screen is responsible for.

The briefing screen establishes the scene before introducing suspects. That ordering is deliberate: reading the crime before meeting the cast means the player approaches suspects with context rather than meeting them cold. The evidence board uses tiered unlocking — Tier 1 clues are available immediately, Tier 2 and 3 unlock as earlier evidence is examined. Players cannot skip to the most incriminating evidence. Investigation has structure.

The reveal gate: 70%, not 100% The killer reveal unlocks when 70% of clues have been examined. Not 100% — that would be punishing, since some clues are genuine dead ends that a good player might deliberately skip. Not 50% — that allows too easy a bypass. 70% requires systematic investigation without demanding completionist behaviour. I set this threshold in the PRD before building the progress ring, which meant the UI was designed around the number rather than the number being adjusted to fit the UI.

Three product problems disguised as UX bugs

Building with AI tools surfaces product problems faster than traditional development because the tool will faithfully implement whatever you specified — including things you specified poorly. Three moments in the build were instructive.

  • 01Suspect profiles closing on second open — Early builds closed Profile A when Profile B was opened. This felt like a UX bug. It was actually an information architecture decision I hadn't made explicitly: should investigation allow cross-referencing, or require sequential examination? Cross-referencing alibis on-screen is deduction. Holding them in memory is recall. Those are different games. Once I made the decision explicitly — multiple profiles open simultaneously, independent close state — the implementation was trivial. The bug was a missing spec.
  • 02Empty space on clue choice screens — Clue modals had three investigative action choices and a significant amount of empty space below them. This is what I started calling a product smell: empty space in a murder mystery game communicates "unfinished." The fix wasn't purely aesthetic — I added atmospherically blended cobweb corner illustrations, a faint bloody footprint trail, and a knife silhouette. The constraint was "atmospheric, not cluttered." That required defining what each decorative element was doing before placing it.
  • 03The loading screen was breaking the experience — AI generation takes 8–15 seconds. Early builds showed a spinner and the word "Generating." This broke the atmospheric frame at the highest-intent moment in the session — right after the player committed to their configuration. The fix was the factoid system: real detective and criminology trivia floating across the screen during generation. The player's first piece of mystery knowledge arrives before the briefing does. Wait time becomes game time. This was a product decision about what the loading state communicates, not a visual polish decision.

What writing the PRD before building actually changed

I wrote a complete PRD before opening Lovable. I've written about why I do this with Bubble Bloom — but Whodunit pushed it further, because the domain (AI-generated game narrative) has higher stakes for specification failures. Vague intentions in a relaxation app produce mediocre outputs. Vague intentions in a mystery game produce an AI that contradicts itself halfway through a case.

  • 01The PRD is the prompt — Every screen described in precise detail in the PRD became a reliable generation prompt with minimal revision. Every screen described vaguely cost three to five revision cycles. "A screen where users examine clues" generates a generic interface. "A grid of evidence cards with atmospheric icons, tiered unlock based on examination progress, and a modal containing full clue text plus three investigative choices that produce AI-generated unique outcomes" generates what I actually wanted.
  • 02The decision log prevented scope creep in real time — The PRD included an appendix of key decisions with rationale. When the temptation arose to add a leaderboard, a sharing prompt, or a session save, I checked the decision log. "No accounts in v1.0 — return play rate will determine whether session persistence justifies Phase 2 investment." The document wasn't planning. It was scope enforcement at the moment of temptation.
  • 03Tool-switching required a shared spec — I used Lovable for primary development and Replit for specific component work. Two tools with different structural opinions about the same codebase is a project management problem, not a technical one. The PRD was the shared source of truth across both. When I switched tools, I didn't lose the thread because the thread was written down.
  • 04Success metrics shaped features backwards — I wrote the success metrics before I wrote the feature list. "AI coherence rating ≥ 4.0/5.0 on exit survey" meant the AI generation system needed an explicit coherence specification, a JSON schema with validation, and retry logic — all before I thought about what the evidence board should look like. Outcome metrics make feature priority obvious in a way that working forward from features never does.

What I'd specify differently

The one thing the PRD underspecified was the investigative choice outcome system. I wrote "each choice produces a unique AI-generated outcome" without defining what "unique" meant in practice — did it mean different text, different information revealed, or different investigation paths? In the build, this produced choice outcomes that varied in text but not always in investigative value. v2.0 will spec three choice outcome types explicitly: one that reveals new information, one that confirms existing evidence, and one that is a deliberate dead end. Players should need to evaluate choices, not just click through them.

AI Resume Evaluator: how I built a zero-cost candidate screening system with semantic scoring, bias detection, and auto-shortlisting.

May 20269 min readProduct Design · Workflow Automation · AI Systems · n8n
⚙️ View the workflow — github.com/saumyavatsacreates A fully importable n8n workflow JSON. 36 nodes. Drop in your JD, point it at a Drive folder, and it evaluates every resume automatically — scoring, ranking, emailing you the Top 5.
View on GitHub →

I built this in n8n — a no-code workflow automation tool where you connect nodes visually, each node doing one thing, passing its output to the next. The workflow ended up at 36 nodes. What I want to document here is not what the system does, but the specific decisions I made at each stage of building it and why those decisions are visible in the node structure.

It starts with a trigger, not a button

The entry point is a Google Drive Trigger node that watches a folder called Resumes every minute. The moment a new PDF lands there, the workflow fires. There is no button to press, no form to submit, no human in the loop at the intake stage. The trigger node is the product decision: hiring pipelines that require manual action at every step lose speed and consistency. If the intake is automatic, everything downstream can be automatic too.

A second trigger — a Form Trigger node — creates a public-facing URL where candidates can self-submit. Name, email, PDF upload. Both triggers feed into the same pipeline through a deduplication gate: a Code node that checks whether the filename already exists in Baserow before proceeding. Running the same resume twice produces the same row, not a duplicate.

The first n8n lesson: every node is a product decision In code, decisions hide inside functions and conditionals. In n8n, every decision is a visible node on the canvas. The deduplication check is a node. The rate limiter is a node. The wait timer between API calls is a node. This forces you to name every decision and make it explicit — which is exactly what a PRD does on paper.

Two parallel paths: caching the JD, processing the resume

The first major architectural decision in the workflow is separating JD processing from resume processing. The JD changes rarely — maybe once per hiring cycle. The resumes change constantly. So there is a Cache JD Text node that stores the JD in n8n's static workflow data after the first extraction. Every subsequent resume run skips the JD extraction entirely and reads from cache. This is why the workflow stays fast and cheap even at volume: the expensive JD processing happens once.

When the JD does change, a second branch fires: a Code node called JD Change Reset that clears the leaderboard, the insight counters, and the cluster scores — so every resume gets re-evaluated against the new JD cleanly. JD change is not a manual operation. It is a handled event.

The scoring node: rule-based first, AI second

The node I spent the most time on is the Build Prompt v2 Code node. It is 180 lines of JavaScript that does two completely separate things. The first half is deterministic: it computes keyword overlap, cosine similarity between JD and resume skill cluster vectors, and a resume quality score based on impact verbs and metric patterns. These numbers are final before the AI is ever called. The second half constructs a prompt that passes those pre-computed scores to Gemini and asks it only for qualitative explanation — strengths, gaps, career trajectory, education fit.

The separation is enforced structurally. Gemini receives the numbers as fixed inputs. The JSON schema it must return has no field for overriding them. The AI layer genuinely cannot change the score — not because of a soft instruction, but because there is no field in the response schema for it to write to.

  • 01Keyword Score — 40 points, computed in the Code node The JD's top 30 non-stopword terms are extracted and checked against the resume text. Each hit contributes proportionally. This runs in under 5 milliseconds per resume — no API call, no cost.
  • 02Cluster Score — 30 points, computed via cosine similarity Eight skill clusters (ML/AI, Data, Backend, Frontend, Cloud/DevOps, Product/BA, Consulting, Finance) are defined as term lists. Both JD and resume are vectorised as term-frequency counts per cluster. Cosine similarity is computed between the two vectors. This is the node that catches candidates whose vocabulary differs from the JD but whose skill set matches — a data engineer who says "ETL in Airflow" when the JD says "data pipeline experience."
  • 03Quality Score — 15 points, computed via regex in the Code node Impact verbs (achieved, shipped, drove, scaled, reduced, led) and metric patterns (numbers followed by %, x, CR, lakh, users, ARR) are counted directly from the resume text. A resume full of "responsible for" language scores low. A resume with quantified outcomes scores high. This runs entirely in the Code node — no AI involved.
  • 04Education Score — 15 points, the only number Gemini produces This is the only score the AI contributes. It receives the degree, field, and certifications from the resume and scores their relevance to the JD on a 0–15 scale. Bootcamps in the right domain count. An MBA in an unrelated field counts less. This is bounded: 0 to 15, no exceptions.

The nodes that failed first

Three nodes caused problems before the workflow ran cleanly.

  • 01The Flag Error Row node had two outputs — Sheets nodes cannot do that An early version of the error path wired the Flag Error Row (a Google Sheets node) to both the Audit Log and the Stop and Error node. Sheets nodes have one output. n8n silently ignored the second wire, leaving Stop and Error disconnected and invisible on the canvas. The fix was a linear chain: Flag Error Row feeds Write Audit Log, which feeds Stop and Error. One output per non-IF node.
  • 02Two node pairs were stacked on identical canvas coordinates The Hiring Insights Aggregator was positioned at the same x,y as the IF Score node. The JD Change Reset node was stacked on the Embed JD node. In n8n, overlapping nodes render as one — so 36 nodes appeared as 34. The fix was moving each pair apart in the JSON's position array. Nothing in the logic changed; only the coordinates.
  • 03All custom node IDs were non-UUID format n8n requires node IDs to be valid UUIDs (xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx). The initial build used readable slugs like embed_jd_skill_clusters. n8n silently dropped or merged nodes with invalid IDs on import. Regenerating all IDs with uuid.uuid4() fixed the import count from 34 to 36.

What the Top-5 Leaderboard Builder node does

After every score, a Code node called Top-5 Leaderboard Builder reads the current leaderboard from n8n's static workflow data, upserts the current candidate by filename, sorts by score descending, and trims to five entries. If the current candidate is in the top five, it sets a flag. An IF node downstream reads that flag and triggers the Gmail SMTP email only when the leaderboard changes. No flag, no email. This means the recruiter receives exactly one email per leaderboard change — not one per resume scored.

Healthcare Billing AI: designing a 3-stage claims automation pipeline that cuts denial risk, flags coding errors, and routes claims in 90 seconds.

May 202610 min readProduct Design · Healthcare · Workflow Automation · AI Systems
⚙️ View on GitHub — Healthcare Billing Automation Full n8n workflow JSON, ROI analysis, and system architecture. 14 nodes. Includes the 3-stage Gemini pipeline, scoring engine, and switch-based routing logic.
View on GitHub →

This workflow started as a single Gemini call that tried to do everything — extract codes, check compliance, estimate denial risk — in one prompt. It fell apart immediately. One large prompt trying to do three jobs produces outputs that are partially right in unpredictable ways. The fix was decomposition: three separate Gemini calls, each with a single clear responsibility, chained through parse nodes that carry structured data forward. That decision is the entire architecture of the workflow.

The trigger and intake chain

The workflow opens with a Google Drive Trigger node polling a Bills folder every minute. When a new file appears, a Download File node pulls the binary. The next three nodes solve a specific problem: Gemini needs plain text, but billing documents are PDFs — often generated by hospital billing systems, sometimes scanned. A Copy Bill to Google Doc node converts the file using Drive's native copy API. An Export Bill as Text node then calls Drive's export endpoint to pull the plain-text version. A Code node validates the extracted text has minimum length before anything expensive runs — a cheap gate that prevents empty or corrupt files from consuming three API calls downstream.

Why Google Doc conversion instead of a PDF parser Drive's native text export handles structured billing PDFs more reliably than a generic PDF-to-text library because billing documents often use complex table layouts that generic parsers misread. The conversion step adds one node and two API calls but produces clean, column-aligned text that the extraction prompt can parse accurately. Removing it saved 200ms and cost three times more in Gemini retries.

Three Gemini nodes, three jobs

Each AI stage is an HTTP Request node calling Gemini 1.5 Flash, followed by a Code node that parses the JSON response and validates the output before passing it forward. The parse node is not optional — it is what makes the chain debuggable. When Stage 2's compliance check fails, the error is isolated to that parse node. The problem is visible, named, and locatable.

  • 01AI: Extract Billable Items — Stage 1 The prompt asks Gemini to identify every medically and financially significant field in the bill: patient identifiers, dates of service, rendering provider NPI, ICD-10 diagnosis codes, CPT procedure codes with quantities and prices, total billed amount, insurance type, and facility classification. The output is a structured JSON object. The prompt explicitly says: do not evaluate, do not flag, do not score — only extract. Mixing extraction with evaluation in one prompt produces both less accurately.
  • 02AI: Code Mapping and Compliance — Stage 2 This node receives Stage 1's structured JSON and evaluates the codes. It looks for unbundling (two codes billed separately that should be one), upcoding (a higher-complexity code than the service warrants), duplicate line items, missing modifiers, and diagnosis-procedure pairing gaps. After this Gemini call, a Validation Layer Code node applies seven deterministic rule checks independently of the AI output — because some compliance rules should be exact, not probabilistic, and the AI should not be able to argue its way past them.
  • 03AI: Denial Risk — Stage 3 The third Gemini call receives both previous outputs and produces a denial probability estimate, a list of specific documentation gaps, recommended pre-submission actions, and an expected reimbursement range. This is the stage that converts a compliance check into an actionable brief. The billing executive does not receive a list of problems — they receive a list of things to do before the claim goes out.

The scoring engine node

After all three AI stages, a Scoring Engine Code node combines the outputs into a 100-point composite: Code Accuracy (30 points from Stage 2), Validation (25 points from the rule engine), Denial Risk (25 points from Stage 3), and Documentation Completeness (20 points from Stage 3's gap count). The score determines which branch a Switch node takes: Auto-Submit above 80, Human Review between 60 and 79, Hold and Correct between 40 and 59, Reject below 40.

Each branch of the Switch node connects to a tailored Gmail node with a different subject, body, and recommended action. The Auto-Submit email says the claim is clean and has been queued. The Hold email names the specific codes that need correction and the documentation gaps that need filling. Same node type, four different outputs, each appropriate to the claim's actual state.

What broke during the build

  • 01The Switch node's branches were orphaned n8n's Switch node creates output branches dynamically. Early in the build, the four routing branches existed in the Switch node's configuration but the downstream Gmail nodes were not wired to them. The canvas showed the Switch node with four output handles and nothing connected. n8n does not error on unconnected outputs — it silently drops the data. Fixing this required clicking each output handle and drawing the wire manually.
  • 02Gemini API free-tier limits hit during testing After the first day of testing, Gemini's free-tier rate limit was exhausted mid-session. The HTTP Request node returned a 429 error that the parse node downstream could not handle — it expected a candidates array and received an error object. The fix was a Groq migration: updating the HTTP node URL to Groq's OpenAI-compatible endpoint (llama-3.3-70b-versatile) and updating the response parsing from candidates[0].content.parts[0].text to choices[0].message.content. One node URL change and one parse node update.
  • 03$json references broke in Run Once for All Items mode The Scoring Engine Code node was set to run once for all items, which changes how $json behaves — it becomes an array, not a single object. Several property lookups that worked in single-item mode silently returned undefined in batch mode. The fix was switching the node to run once per item, which restored standard $json behaviour throughout.

The Insights Generator node

After every claim is processed, a Code node called Insights Generator updates running totals stored in n8n's static workflow data: total claims processed, average score, score distribution by tier, most common denial risk flags. These aggregate without a database — n8n's static data persists across workflow runs on the same instance. The limitation is that static data resets on redeploy. For anything beyond a few hundred claims, this needs a persistent store. The redesign blueprint in the GitHub document replaces this with BigQuery.

Automation Case Study · Personal Project

Personal Cookbook Engine: one keyword in, a structured recipe out — how I automated the conversion of cooking videos into a searchable personal cookbook.

May 20267 min readAutomation · Product Design · LLM Pipelines · n8n
⚙️ View on GitHub — Personal Cookbook Automation Full n8n workflow. 11 nodes. Enter a dish name, get a clean structured recipe appended to your Google Docs cookbook. 30 seconds. Zero manual steps.
View on GitHub →

This is the smallest workflow I have built — 11 nodes — and the one that taught me the most about what a node actually is. Not a step. A decision made visible. Every node in this workflow exists because I made a specific product choice that needed to be represented somewhere in the chain. The 11 nodes are 11 decisions. Here is what they are and why I made them.

Node 1 and 2: Manual Trigger and Set Keyword

The workflow starts with a Manual Trigger node — you run it when you want a recipe, not on a schedule. The second node is a Set node that stores the search keyword as a named variable: keyword equals "paneer butter masala recipe." This is not a technical requirement. n8n would let you hardcode the keyword inside the HTTP Request node downstream. I put it in a Set node because variables that live in a dedicated node are visible, editable, and nameable without touching any other node in the chain. When you want to change what recipe you are looking for, you change one value in one node.

Nodes 3 and 4: YouTube Search and Video ID Extraction

Node 3 is an HTTP Request node calling the YouTube Data API v3 search endpoint with the keyword, returning the top 5 video results with their IDs, titles, and channel names. Node 4 is a Code node that extracts only the videoId field from each result and splits them into separate pipeline items — one item per video. This is the first time the pipeline fans out from one item to five.

The reason for a dedicated Code node here rather than using an expression in the next node is debuggability. When I click Node 4's output in the n8n editor, I can see exactly what five items are flowing forward and inspect each videoId. If the YouTube API returns a malformed response, the error surface is isolated to this node.

Nodes 5 and 6: Statistics Fetch and Popularity Filter

Node 5 is another HTTP Request node — this time calling the YouTube Videos API to fetch viewCount for each video ID. Node 6 is an IF node with one condition: viewCount greater than 100,000. Videos that fail this check are discarded. Only qualifying videos proceed.

Why an IF node instead of an LLM call The first version used an AI node to classify whether a video title suggested a reliable recipe. It was slower, more expensive, and less consistent than a number threshold. 100,000 views is a proxy for quality validated by audience behaviour at scale — a deterministic signal that costs nothing. When you can replace a probabilistic AI judgement with a deterministic rule, you should. The IF node is one of n8n's most underused tools for exactly this reason.

Nodes 7 and 8: Transcript Extraction and Parsing

Node 7 is an HTTP Request node calling the RapidAPI YouTube Transcript API with the videoId. The response is an array of objects — each object has a text field containing a sentence or phrase from the video. This structure is what the API returns; it is not useful yet.

Node 8 is a Code node that joins all the text fields with a space into a single raw string. One input (array of objects), one output (single string). The entire node is three lines of JavaScript. It exists as a separate node because the LLM prompt in Node 9 expects a single string, and the transformation from array to string is a named, visible, testable step rather than an expression buried inside the prompt.

Nodes 9 and 10: Two LLM calls, two jobs

Both AI nodes call Gemini 1.5 Flash via HTTP Request. They are separate nodes with separate prompts and separate responsibilities.

  • 01Node 9 — Recipe Generation The prompt receives the raw transcript string and asks Gemini to extract a structured recipe: ingredient list with quantities and units, ordered steps, serving size, prep time, cook time. The output must be a JSON object. The prompt says nothing about formatting — only extraction. Hindi cooking videos arrive as unpunctuated Hinglish with filler words and host asides. The prompt handles this by asking for ingredients and steps only, not a transcript summary.
  • 02Node 10 — Recipe Formatting The second prompt receives Node 9's structured JSON and asks Gemini to rewrite it as a clean, readable recipe in natural language: consistent terminology, numbered steps, ingredient quantities in standard units. It is not asked to re-extract anything. Keeping the two calls separate means formatting errors are isolated to Node 10 and extraction errors are isolated to Node 9. If the formatted recipe looks wrong, you check Node 10's input — if Node 9's output was clean, the formatting prompt needs work. If Node 9's output was already garbled, the extraction prompt needs work. Mixing them in one call makes this distinction impossible.

Node 11: Google Docs Append

The final node is a Google Docs node that appends the formatted recipe to a named document. The document functions as a personal cookbook — every recipe runs add a new entry at the end. This is append-only, which is a deliberate scope decision: search, indexing, and cuisine categorisation were all considered and all cut. Each would have added a node, an API dependency, and a new failure surface. The core value is a recipe in a findable place. Everything else is v2.

The one node I removed

An early version of the workflow had a Telegram notification node at the end that sent a message when a recipe was added. It was the first thing I cut. It added a Telegram API credential, a new failure point, and a notification for something that already produced a visible artefact — the recipe in the doc. Notifications that confirm a thing you can already see are noise. The removal was a product decision, not a technical one: the pipeline's output is the doc. The doc is the notification.
